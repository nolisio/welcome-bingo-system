# Welcome Bingo System 実装仕様書

## 1. 本書の位置付け

本書は、このリポジトリにおける今後の実装・改修の基準となる実装仕様書である。

対象は「新歓で1回運用するためのシステム」であり、過度な将来拡張よりも、当日に安定して運用できることを優先する。

- 想定用途: 新歓イベント用
- 想定規模: 参加者約60名
- 想定運用: 1会場、1ゲーム、1つの司会進行
- 重視する点: 操作のわかりやすさ、ラグの少なさ、当日の事故防止、実装の単純さ

## 2. 対象範囲

### 実装対象

- 参加者用スマホ画面
- 管理者用画面
- 会場投影用画面
- リアルタイム投票
- ビンゴ進行
- 参加時の新入社員判定
- 新入社員のみ中央マス初期オープン
- 新入社員による作問
- 管理画面での作問承認フロー
- 単一サーバー・単一ゲームでの運用

### 実装対象外

- 景品管理
- 当選者と景品の紐付け
- 複数ルーム対応
- イベント管理システム化
- 重い監査ログ機能
- 将来の汎用化を前提にした過剰設計

## 3. システムの前提

本システムは、単一グローバルゲーム方式で実装する。

ここでいう単一グローバルゲームとは、サーバー上で進行中のゲーム状態を1つだけ持ち、全参加者が同じゲームに参加する方式を指す。

- 参加者は全員同じゲームに参加する
- 管理画面の操作は全参加者に対して効く
- 会場表示画面も同じゲームを表示する
- ルーム切り替えや複数ゲーム同時進行はしない

QRコードは参加画面を開くための導線として使うが、アプリ内部ではルームコード管理を前提としない。

## 4. 採用技術方針

- フロントエンド: Next.js + TypeScript
- バックエンド: Node.js + Express + Socket.IO
- データベース: PostgreSQL + Prisma
- 進行中の状態管理: サーバーメモリ

現行コードベースの方向性と一致しているため、この方針を維持する。

## 6. ゲーム進行ルール

1. 参加者はURLまたはQRコードから参加画面を開く
2. 参加者は表示名を入力する
3. 参加者は新入社員かどうかを `はい / いいえ` で回答する
4. サーバーは参加者を登録し、ビンゴカードを配布する
5. 新入社員であれば中央マスを初期オープンにする
6. 新入社員でなければ中央マスは初期オープンしない
7. 管理者がゲームを開始する
8. 各ラウンドで質問と選択肢A/Bを提示する
9. サーバーが未使用数字を1つ抽選する
10. 参加者がA/Bのどちらかに投票する
11. 管理者が投票を締め切る
12. サーバーが多数派を判定する
13. 多数派に投票し、かつ自分のカードに抽選数字を持つ参加者のみ、そのマスを開く
14. サーバーがビンゴ判定を行う
15. 管理者が終了またはリセットするまでこれを繰り返す

## 7. ビンゴカード仕様

- 5x5の25マス
- B/I/N/G/O の列レンジに従って 1〜75 を割り当てる
- 中央マスは通常の番号マスとする
- 中央マスを全員共通のフリー扱いにはしない
- 新入社員のみ、中央マスを初期オープン状態で開始する

これは重要な仕様である。新入社員優遇を成立させるため、中央マスは「全員共通のフリー」ではなく、「新入社員だけが最初から開いている通常マス」とする。

## 8. 多数派判定と同数時の扱い

- 多数派は得票数が多い側とする
- A票とB票が同数の場合、多数派なしとする
- 多数派なしの場合、そのラウンドでは誰のマスも開かない

設計書の拡張案としてのランダム決着は採用しない。今回のイベント用途では、仕様が単純で説明しやすいことを優先する。

## 11. 参加フロー

参加フローは以下とする。

1. 名前を入力する
2. 「あなたは新入社員ですか？」に `はい / いいえ` で回答する
3. その2つをまとめて参加要求として送信する
4. 参加者状態とカードを受け取る

新入社員判定は今回のイベントでは自己申告でよいものとする。

運用上の方針:

- 初期選択は `いいえ`
- 管理者画面から新入社員フラグを確認できるようにする
- 将来的に必要であれば、管理者による修正機能を追加してよい

## 12. 新入社員作問と承認フロー

新入社員作問は今回の実装対象であり、承認フロー付きで実装する。

### 承認フローの意味

新入社員が送信した質問を、そのまま即時公開せず、管理者が内容を確認してから採用する流れを指す。

### 基本フロー

1. 管理者が対象の新入社員を選ぶ
2. 対象の新入社員だけに簡易作問フォームを表示する
3. 新入社員が質問文、選択肢A、選択肢Bを送信する
4. サーバーはその作問を `PENDING` で保存する
5. 管理者画面で作問内容を確認する
6. 管理者は以下のいずれかを行う
   - そのまま承認
   - 軽く修正して承認
   - 却下
7. 承認済みの質問だけが次のラウンドで使用可能になる

### このフローを入れる理由

- 不適切表現の防止
- 長すぎる質問やわかりにくい質問の防止
- 当日の進行停止防止
- 最終責任を司会側に残すため

### 対象条件

- `isNewEmployee = true` の参加者のみが作問対象となる

## 13. リアルタイム通信方針

リアルタイム処理は Socket.IO を主とする。

### 既存互換で維持するイベント

- `participant:join`
- `participant:reconnect`
- `vote:submit`
- `public:subscribe`
- `admin:start-game`
- `admin:start-round`
- `admin:close-voting`
- `admin:reset-game`

### 追加実装するイベント

- `custom-question:open`
- `custom-question:submit`
- `custom-question:submitted`
- `admin:custom-question:list`
- `admin:custom-question:approve`
- `admin:custom-question:reject`

イベント名は実装時に多少調整してよいが、振る舞いは本仕様に従う。

## 14. REST API方針

REST API は最小限に留める。

主な用途:

- ヘルスチェック
- 現在の公開ゲーム状態取得
- 管理画面向け参加者一覧取得
- 必要であれば管理画面向け作問一覧取得

今回のイベント向け実装では、ルームベースの REST API 設計は採用しない。

## 15. 現行コードとの整合方針

現行コードと整合している点:

- Next.js フロントエンド
- Express + Socket.IO バックエンド
- PostgreSQL + Prisma
- サーバーメモリによる単一ゲーム状態管理

現行コードから変更すべき点:

- `isNewEmployee` の追加

## 16. 実装時の記述ルール

### 文字コード

- 新規作成する Markdown、TypeScript、TSX、設定ファイルは UTF-8 で扱う
- ユーザー向け日本語文言は UTF-8 の日本語で記述する
- 文字化けした文言は見つけ次第修正対象とする

### コードコメント方針

今後の実装では、必要な箇所に短く意図がわかるコメントを残す。

コメントを残すべき箇所:

- 新入社員優遇ルール
- 中央マスの扱い
- 多数派判定や同数時処理
- 作問承認フロー
- 再接続復元の意図
- 当日運用で誤解されやすい仕様分岐

コメントを残さなくてよい箇所:

- 変数代入など自明な処理
- 関数名を読めばわかる処理
- JSX の単純な表示部分

コメントの書き方:

- 長文ではなく短く書く
- 「何をしているか」より「なぜそうしているか」を優先する
- 実装依存の細かすぎる説明ではなく、業務仕様や意図を残す
- 必要に応じて日本語コメントを使ってよい

## 18. 完了条件

以下を満たしたとき、本仕様に対して実装完了とみなす。

- 参加者が名前と新入社員回答を入力して参加できる
- 再接続時に同一参加者として復元できる
- 新入社員だけ中央マスが初期オープンになる
- 新入社員以外は中央マスが初期オープンにならない
- ラウンド開始、投票、結果確定がリアルタイムで進行できる
- 多数派判定とマス開放が仕様どおり動作する
- ビンゴ判定が正しく動作する
- 指名された新入社員が作問できる
- 管理者がその作問を承認または却下できる
- 景品管理がなくても当日運用できる

## 19. 基本原則

このプロジェクトでは、「理論上の拡張性」より「新歓当日に確実に回ること」を優先する。

設計判断で迷った場合は、以下を優先する。

- 実装が単純であること
- 司会が迷わないこと
- 参加者が迷わないこと
- 当日に壊れにくいこと
- 修正しやすいこと

将来の複数イベント対応や汎用システム化は、本イベントが無事に運用できた後にあらためて検討する。
